## 1. 프로젝트 배경

### 1-1. 연구 배경

복합 만성질환 CKD(콩팥병)·DM(당뇨병) 환자는 지속적으로 증가하고 있으며, 많은 환자와 간병인이 이미 ChatGPT 등 범용 LLM을 통해 검사 수치 해석, 복약 상호작용, 식이 관리, 병원 탐색, 복지·보험 정보를 스스로 찾고 있다. 기존 일반 LLM+RAG 챗봇은 의료 문서를 검색해 답변을 생성할 수 있으나, 한국어 복합 만성질환 질의에서는 여전히 정확도 한계가 크다. 환자식 한국어 표현을 표준 개념으로 안정적으로 정규화하지 못하고, 복합 질환·검사수치·복약·생활관리·지원제도 사이의 관계를 구조화하지 못한 채 유사 문장 중심으로 검색하기 때문이다. 또한 실제 사용자 질문은 필요한 정보가 생략된 불완전 질의인 경우가 많아, 누락 정보를 확인하지 않은 상태에서 생성된 답변은 범용적이거나 부정확해지기 쉽다.

본 연구의 초점은 한국어 복합 만성질환 RAG 챗봇에 대비해서 Ontology와 LoRA가 각각 어떤 층위의 문제를 개선하는지를 분리해 검증하는 데 있다.

### 1-2. 대표 질의 예시

- “약 먹고 어지럽고 식은땀 나는데 저혈당인지 신장 때문인지 모르겠어요” → 증상 해석과 응급도 판별이 동시에 필요한 질의
- “투석 얘기 들었는데 회사 다녀서 밤에 가능한 데가 있나” → 질환 진행도, 치료 방식, 병원 탐색, 생활 조건이 함께 들어간 질의
- “당도 오락가락하고 발끝이 찌릿한데 운동 계속 해도 되나요?” → 혈당 변동, 신경병증, 운동 가이드가 결합된 질의
- “콩팥 안 좋은데 산정특례 되는 단계인가요, 아직 투석은 안 해요” → 질환 단계와 보험·복지 자격 판단이 함께 필요한 질의

### 1-3. 연구 목적

본 프로젝트는 일반 RAG, RAG+Ontology, RAG+Ontology+LoRA Fine-tuning 3개 조건을 동일한 서비스 흐름과 동일한 공통 가이드 아래 비교하여, Ontology와 LoRA가 각각 관계 추론 강화와 한국어 의료 표현 이해 개선에 어떤 기여를 하는지 분리 검증한다.

## 3. 제안 해결책 

### 3-1. 실험 비교 그룹 정의
```
Group A: RAG
Group B: RAG + Ontology
Group C: RAG + Ontology + LoRA Fine-tuning
```

### 3-2. 수평 비교 원칙

세 그룹은 동일한 사용자 입력, 동일한 Intent Agent, 동일한 Profile Check, 인터페이스를 공유한다. 공통 파이프라인 안에서 Ontology Reasoning과 LoRA Medical Understanding이 추가될 때 의미 해석과 검색 품질이 어떻게 달라지는지 측정하는 것이다.

연구 통제를 위해, 3개 모델 모두 다음의 공통 흐름을 동일하게 따른다.
로그인 사용자 프로필 로드
사용자 발화 수신
의도 분류(증상 문진 / 병원 정보 등)
멀티턴 응답 진행
응답 출력
평가 지표 자동 기록
모델 간 차이는 이 상위 흐름을 변경하지 않고, 슬롯·지식 해석 방식에서만 발생하도록 설계한다.

### 3-3. 각 계층의 역할

- **Intent Agent**: 질문을 증상 문진, 병원 정보, 복지·보험, 식이·운동, 연구·신약 등 적절한 에이전트 범주로 분류
- **Profile Check**: 로그인 사용자의 질환 프로필, 투석 여부, 약물, 최근 수치 등 저장된 컨텍스트를 공통 반영
- **Ontology Seasoning**: 질환, 증상, 검사수치, 치료, 합병증, 지원제도 간 관계를 명시 규칙으로 확장
- **LoRA Medical Understanding**: 한국어 환자식 구어체, 생략 표현, 비정형 질의의 의학적 의미 파악을 정밀 보정
- **Slot Check**: 각 에이전트가 답변 전 반드시 확인해야 할 필수 정보 존재 여부를 공통 규칙으로 점검 (클로드의 입력창 위 3-4개 체크박스 프리셋/슬롯 체크)
- **Search**: VectorDB, 규정 문서, 병원 DB, 복지 정보 DB 등 범주별 데이터 소스 검색
- **Answer**: 근거 기반 자연어 응답 생성 및 비교 지표 출력

## 4. 범위 및 기능 정의

### 4-1. 서비스 범위 (우선순위)

시간 제약상 초기 구현과 시연은 증상 문진 및 병원 정보를 우선으로 한다.  연구 논리와 데이터 구조는 아래 범주 확장을 전제로 설계한다.
- 증상 문진
- 병원 정보
- 복지·보험
- 식이·운동
- 연구·신약

### 4-2. 핵심 기능

- 로그인 기반 사용자 프로필 연동형 AI 챗봇
- 동일 질의에 대한 3개 그룹 병렬 응답 비교
- Intent Agent 기반 범주 라우팅
- Ontology Reasoning 기반 관계 확장
- LoRA Medical Understanding 기반 한국어 의료 구어체 보정
- RAG 근거 검색 및 출처 기반 응답 생성
- 평가 지표 자동 수집 및 대시보드 시각화

## 6. 공통 워크플로우 설계

### 6-1. 공통 처리 규칙

- 모든 그룹은 Intent Agent를 먼저 통과한다.
- 모든 그룹은 Profile Check를 공통 수행한다.
- 모든 그룹은 Slot Check를 수행한 뒤 Search로 넘어간다.
- 그룹별 차이는 Slot Check 자체가 아니라, 그 전에 들어가는 의미 이해 계층(LoRA)과 관계 추론 계층(Ontology)의 존재 여부다.

### 6-2. Ontology 기반 동적 슬롯 필링 데이터 모델

온톨로지(예: Neo4j) 상에 데이터를 주입하여 챗봇 UI에 동적으로 슬롯 수집 질문 및 다지선택 옵션을 연동하기 위한 구조는 다음과 같다.

#### 1) 노드 및 속성 정의
*   `(:Intent)`: 사용자의 대화 목적 (예: `name: "HospitalSearch"`)
*   `(:Slot)`: 의도 해소를 위해 필요한 속성 (예: `name: "DialysisType"`, `prompt: "어떤 투석 치료를 받고 계신가요?"`)
*   `(:Option)`: 슬롯 충족을 위해 UI에 표시할 선택지 (예: `value: "혈액투석"`, `sort_order: 1`)

#### 2) 관계 정의
*   `(:Intent)-[:REQUIRES]->(:Slot)`: 특정 의도가 활성화되었을 때 반드시 수집되어야 하는 필수 속성을 연결한다.
*   `(:Slot)-[:HAS_OPTION]->(:Option)`: 각 슬롯이 보유한 선택 항목 목록을 연결한다.

#### 3) Neo4j 데이터 주입 구문 예시 (Cypher)
```cypher
// 의도 및 필수 슬롯 생성
MERGE (i:Intent {name: "HospitalSearch"})
MERGE (s1:Slot {name: "Location", prompt: "찾으시는 위치를 선택해 주세요."})
MERGE (s2:Slot {name: "DialysisType", prompt: "어떤 투석 치료를 받고 계신가요?"})

// 필수 관계 연결
MERGE (i)-[:REQUIRES]->(s1)
MERGE (i)-[:REQUIRES]->(s2)

// 슬롯에 대한 선택지(Option) 매핑
MERGE (o1:Option {value: "혈액투석", sort_order: 1})
MERGE (o2:Option {value: "복막투석", sort_order: 2})
MERGE (o3:Option {value: "투석 안 함", sort_order: 3})

MERGE (s2)-[:HAS_OPTION]->(o1)
MERGE (s2)-[:HAS_OPTION]->(o2)
MERGE (s2)-[:HAS_OPTION]->(o3)
```

#### 4) 백엔드 주입 API 반환 JSON 규격
의도 탐지 후, RAG 검색 전 필수 슬롯 정보 누락을 감지했을 때 챗봇 UI의 슬롯 상자로 반환하는 데이터 구조이다.
```json
{
  "slot_status": "pending",
  "missing_slot": "DialysisType",
  "slot_index": {
    "current": 1,
    "total": 2
  },
  "ui_schema": {
    "prompt_text": "어떤 투석 치료를 받고 계신가요?",
    "options": [
      { "id": 1, "label": "혈액투석" },
      { "id": 2, "label": "복막투석" },
      { "id": 3, "label": "투석 안 함" }
    ],
    "allow_skip": true,
    "allow_custom": true
  }
}
```

### 7-1. 핵심 지표

## 9. 제한 사항 및 가드레일 정책

- **의료 서비스 Out-of-Scope 제한 정책**: 
  - 콩당콩당 AI 챗봇 서비스는 만성 신장병(CKD), 당뇨병(DM), 고혈압(HTN) 3대 핵심 타겟 분야에 대해서만 RAG 기반 답변을 제공합니다.
  - 비의학적 일반 주제(예: 날씨, 주식, 연예, 프로그래밍 등) 및 타겟 질환군 외의 의료 주제(예: 암, 코로나, 감기, 임플란트 등)는 차단 프로토콜(`safetyChecker.ts`)을 통해 즉각 감지하여 표준 거절 응답을 반환합니다.
- **의료 가드레일 규정**:
  - 임의적인 진단 및 확진 행위를 절대 수행하지 않으며, 처방전 및 진단서 작성을 금지합니다.
  - 환자가 임의로 전문 의약품 복용량을 조절하거나 약물을 무단 변경/중단하도록 유도하는 질의에 대해서는 복용 변경의 위험성을 경고하고 차단합니다.
  - 모든 의학 관련 응답 하단 또는 맥락에 담당 주치의와의 대면 진료 및 처방 우선 지침과 면책 문구를 굵은 글씨로 필수 노출합니다.
- AI 챗봇은 로그인 사용자만 사용 가능
- 본 서비스는 의학적 진단·처방을 대체하지 않음
